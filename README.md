## 为什么需要这个插件？

> 当你在群里问 AI "为什么清华开设了原神课？"，AI 会一本正经地开始胡说八道。

传统的 LLM 对话流程有一个致命盲区：**它不会质疑问题本身的前提是否成立**。这个问题在群聊环境中尤为严重——

### 场景一：AI 被带节奏

> **群友**：为什么银价比金价还贵？  
> **AI**（没有该插件）：银价高于金价的原因有很多，首先从供需关系来看……

AI 没有发现问题的预设前提是错误的。它基于一个虚假前提，生成了一段看起来"专业"但完全错误的内容。

### 场景二：事实性陷阱

> **群友**：咸菜为什么不加防腐剂？  
> **AI**（没有该插件）：咸菜传统工艺通过高盐分抑制细菌，因此不需要额外添加防腐剂……

用户用"为什么不"的句式预设了"不加防腐剂"为事实。但现实是许多工业化咸菜确实会添加防腐剂。AI 被诱导产生了幻觉。

### 场景三：人类本质是复读机

> **群友 A**：为什么清华开设了原神课？  
> **群友 B**：为什么北大开设了原神课？  
> **群友 C**：为什么复旦开设了原神课？  
> **AI**：（被迫回答三次内容几乎一样的废话）

## 核心功能

### 全能预审 (Unified Check)

单次 LLM 调用同时完成 **意图判定 → 前提提取 → 事实核查**，将传统串行 2-3 次 API 调用压缩为 1 次：

```
用户消息 → [单次 LLM] → JSON {is_factual_question, premises[], premise_truths[], premise_relation, corrections[], logic_flaw, needs_search}
                        ↓
                    多维审计 / 放行
```

支持 **「普通提问」与「预设前提提问」** 的精准区分：

| 用户消息 | 判定 | 原因 |
|---|---|---|
| 咸菜会加防腐剂吗？ | 普通提问 → 放行 | 开放性问题，无预设前提 |
| 咸菜为什么不加防腐剂？ | 预设前提 → 核查 | 隐含"不加防腐剂"这一虚假前提 |
| 为什么天空是蓝色的？ | 预设前提 → 通过 | 隐含"天空是蓝色的"，但前提正确 |
| 北大和清华都开了原神课！ | 多前提 → 核查 | 拆解为两个独立前提逐一核证 |
| 清华或北大开了原神课？ | OR 关系 → 仅全假时纠错 | 选择关系下任一成立即可 |

### 多维事实审计 (Multi-Premise Audit)

不再仅限于简单句，能够将复杂句式拆解为 **原子化前提列表** 并逐一核证：

- **AND 关系**（并列、递进、因果）：**一假即纠** — 只要任一前提为假即触发纠错
- **OR 关系**（选择）：**全假才纠** — 仅当所有前提均为假时才触发纠错
- **XOR 关系**（排他或）：**非一即纠** — 恰好一个为真时成立，否则纠错
- 多前提纠错文案自动聚合为编号列表，清晰展示每个错误及对应纠正

### 🔥 实验性：对线模式 (Argumentative Mode)

针对逻辑发烧友的终极审查模式。开启 `enable_argumentative_mode` 后，插件注入 **逻辑学专家人格**，除事实核查外还深度批驳论证结构：

- 识别 **XOR（异或）、IMPLICATION（蕴含）、BICONDITIONAL（双条件）** 等复杂逻辑关系
- 检测 **不当类比、非黑即白、诉诸无知、滑坡谬误、因果倒置、稻草人谬误** 等常见逻辑谬误
- JSON 输出新增 `logic_flaw` 字段，专门描述逻辑推导中的谬误
- 纠错文案聚合为 **事实纠正 + 逻辑批驳** 组合输出

```
❌ 1. 北大开设了原神课
   2. 清华开设了原神课

✅ 修正：
1. 北大没有开设原神课
2. 清华也没有开设原神课

⚠️ 逻辑漏洞：不当类比 — 不能因为两所学校同为顶尖学府就推导出它们开设相同课程
```

### 异步乐观执行

开启 `enable_async_mode` 后，插件在后台异步核查，完全不阻塞 AstrBot 的主回复流程：

```
主线程：  消息进入 → 立即放行 → AI 生成回复（零延迟）
后台任务：              消息进入 → LLM 核查 → 判定结果
                                     ├─ 跑赢 → stop_event 拦截
                                     └─ 跑输 → 追击纠错 / 闪电撤回
```

### 智能跟风拦截

基于 **difflib SequenceMatcher** 的语义相似度算法，精准捕捉"换词复读"：

- 支持 `algorithm`（纯算法，极速）和 `llm`（算法初筛 + LLM 二次确认，高准）两种模式
- 跟风命中后支持 `intercept`（拦截）和 `check_anyway`（吐槽后继续核查）两种策略

### 闪电撤回

当异步模式下核查发现错误、但 AI 回复已经发出时，插件会：

1. 尝试通过 OneBot 协议 **撤回 AI 的错误回复**（仅 QQ，需管理员权限）
2. 撤回成功 → 发送纠错消息（前缀：`recall_success_prefix`）
3. 撤回失败 → 发送追击更正（前缀：`recall_fail_prefix`）
4. 发送失败 → 静默兜底，不崩溃

## 安装与配置

### 安装

1. 将 `presupposition_validation` 目录复制到 AstrBot 的 `data/plugins/` 目录下
2. 重启 AstrBot，在 WebUI 的 **插件管理** 中启用本插件
3. 根据需要调整配置项

如需联网搜索功能，请额外安装可选增强依赖（插件启动时会自动探测）：

```bash
pip install duckduckgo-search
```

> 插件会自动探测运行环境：若已安装 `duckduckgo-search`，将启用独立搜索；若框架中已注册其他搜索工具（如 Tavily、SerpAPI 等插件），将优先使用框架原生搜索能力；若无任何搜索来源，将降级为纯 LLM 常识校验，不影响核心功能。

### 核心配置项

<details>
<summary><b>⚙️ 基础设置</b></summary>

| 配置项 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `enabled` | bool | `true` | 总开关，关闭后所有消息直接放行 |
| `api_request_timeout` | int | `15` | 所有 LLM/网络请求的超时秒数 |
| `enable_async_mode` | bool | `false` | 【实验性】异步并行核查，降低闲聊延迟 |
| `enable_recall` | bool | `false` | 异步模式下尝试撤回错误消息（需管理员权限） |
| `enable_meme_detect` | bool | `true` | 开启群聊跟风造句检测 |
| `enable_argumentative_mode` | bool | `false` | 【实验性】对线模式，注入逻辑学专家人格，识别 XOR/蕴含等复杂逻辑谬误 |
| `history_window_size` | int | `5` | 跟风检测的历史消息窗口大小 |
| `similarity_threshold` | float | `0.75` | 跟风检测相似度阈值（0~1） |

</details>

<details>
<summary><b>🧠 策略模式</b></summary>

| 配置项 | 可选值 | 默认值 | 说明 |
|---|---|---|---|
| `fact_check_method` | `llm` / `web_search` | `llm` | 事实核查方式 |
| `action_mode` | `intercept` / `warn_and_answer` | `warn_and_answer` | 发现虚假前提时的处理方式 |
| `meme_detect_method` | `algorithm` / `llm` | `algorithm` | 跟风判定算法 |
| `meme_response_mode` | `fixed` / `dynamic_llm` | `dynamic_llm` | 跟风吐槽回复方式 |
| `meme_action_mode` | `intercept` / `check_anyway` | `intercept` | 跟风命中后的放行策略 |

</details>

<details>
<summary><b>💬 文案模板</b></summary>

所有用户可见的文案均可通过 WebUI 自定义，**留空则静默不发送**：

| 配置项 | 说明 |
|---|---|
| `intercept_message_text` | 拦截纠错消息，可用 `{premise}` `{correction}` |
| `warning_prefix_text` | warn_and_answer 模式注入 system_prompt 的提示 |
| `correction_followup_text` | 异步追击纠错正文 |
| `meme_followup_text` | 异步追击跟风拦截，可用 `{original}` `{copy}` |
| `recall_success_prefix` | 撤回成功前缀 |
| `recall_fail_prefix` | 撤回失败前缀 |
| `meme_async_roast_prefix` | 异步追击吐槽前缀，可用 `{roast}` |
| `meme_fixed_reply_text` | 固定吐槽文案 |
| `error_fallback_text` | 异常兜底文案 |

</details>

## 技术架构

```
用户消息
  │
  ├─ enabled? ──→ 放行
  ├─ len < 3? ──→ 放行
  │
  ├─ enable_async_mode = true?
  │   ├─ Yes → create_task(管线) → 立即放行
  │   └─ No  → await 管线
  │
  ▼
单次 LLM 全能预审
  ├─ is_factual_question = false → 放行
  ├─ enable_meme_detect → 跟风检测（原始消息 + 前提级比对）→ 吐槽/拦截
  ├─ 多前提审计
  │   ├─ AND：任一 false → 纠错
  │   ├─ OR：全部 false → 纠错
  │   └─ XOR：非恰好一个 true → 纠错
  ├─ needs_search = true → 联网搜索兜底
  └─ logic_flaw ≠ "" → 逻辑批驳（事实纠正 + 谬误指出）
```

- **语言**：Python 3.10+
- **依赖**：`pydantic >= 2.0.0`、`duckduckgo-search >= 6.0.0`（可选）
- **框架**：AstrBot Plugin SDK (`astrbot.api`)
- **代码量**：~1000 行核心逻辑

## 免责声明

> [!WARNING]
> 本插件的核查结果完全依赖于底层 LLM 的常识能力与联网搜索结果。
> **不保证 100% 准确，可能存在误判或漏判的情况。**
> 
> 建议将 `action_mode` 设为 `warn_and_answer`（默认值），在指出错误的同时保留 AI 回答能力，
> 以避免对正常对话造成过度干扰。对于 `enable_async_mode` 和 `enable_recall` 等实验性功能，
> 请在充分测试后再在生产环境中启用。
